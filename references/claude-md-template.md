### raw/ 与 research/ 的区别

- **raw/** = 你看到好内容，主动收藏的 → 人类策展，只读

## 知识管理方法论

本知识库融合三种方法论，形成完整的知识闭环：

### Zettelkasten（卢曼卡片盒）

**三层笔记体系**：
1. **闪念笔记（Fleeting）** → raw/ 中的随手记录
2. **文献笔记（Literature）** → wiki/ 中的原子笔记（/compile 产出）
3. **永久笔记（Permanent）** → wiki/_MOC/ 中的 MOC 和主题总结（/summarize 产出）

**原子性原则**：
- 每篇笔记只讲一个概念/洞察/模型
- 一份 raw/ 材料涉及多个概念 → 拆分为多篇原子笔记
- 多份材料讨论同一概念 → 合并到同一篇笔记

### MOC（Map of Content）

**导航枢纽**：
- 每个主题有一个 MOC，链接该主题下所有相关笔记
- MOC 提供学习路径建议
- 顶层 MOC（知识总览）链接所有主题 MOC
- 文件夹负责物理组织，MOC 负责逻辑导航

### Dataview + Tasks 插件集成

**Tasks 插件语法**（在笔记中标记任务）：
\```markdown
- [ ] 任务描述 #todo 📅 2026-07-05
- [x] 已完成任务 ✅ 2026-07-03
\```
- `📅` 截止日期 / `🔼` 计划日期 / `🔁` 循环 / `✅` 完成 / `#todo` 过滤标签

**Dataview 查询**（在笔记中嵌入动态视图）：
\```
TABLE type, updated
FROM "wiki"
WHERE type = "concept"
SORT updated DESC
\```

## 编译哲学

这个知识库是自下而上生长的。wiki 的结构、分类、重点方向完全由 raw/ 中的实际材料决定，不预设任何主题框架。

- 不要假设我关注什么方向，让材料自己说话
- 如果 raw/ 中某个话题材料多，wiki 中那个方向自然就厚
- 编译时忠于材料本身的内容，不要为了"全面性"去填充你自己的知识
- 分类和索引是对已有内容的归纳，不是预先搭好的框架等着填

### 关于 INDEX.md 与多层级文件夹结构

在 wiki/ 根目录维护 `INDEX.md`，每次编译和整理后更新。

**多层级规则**：
1. wiki/ 下最多 3 层文件夹深度（`wiki/主题/子主题/` ）
2. 文件夹名称用中文，专有名词保留英文
3. 每个叶子文件夹至少 2 篇文章，否则合并到上级
4. 当主题内容增多时，可以拆分子主题（动态演化）

6. `[[]]` 链接中使用文件名即可，Obsidian 会自动解析路径
7. 一篇文章只放在它最主要的分类文件夹中
8. `_compile-log.md`、`_changelog.md`、`INDEX.md`、`_MOC/` 放在 wiki/ 根目录

## 十个核心命令

### /compile — 编译命令

从 raw/ 和 research/ 提取原子洞察、概念、模型到 wiki/。

**只负责提取，不负责整理**：
- ✅ 提取核心洞察、可复用模型、关键概念
- ✅ 拆分多主题材料为独立原子笔记
- ✅ 建立笔记间的双向链接
- ❌ 不做主题级总结（/summarize 的职责）
- ❌ 不创建 MOC（/summarize 的职责）

### /summarize — 整理命令

组织 wiki/ 中的原子笔记，创建 MOC 和主题总结。

- 分析主题聚类
- 创建/更新多层级文件夹结构
- 创建 MOC 导航笔记
- 创建主题总结笔记
- 补充双向链接
- 更新 INDEX.md

### /todo — 日记整理命令

整理今日待办、计划和学习情况，写入当天日记。支持多次执行合并去重。

### /lint — 格式检查命令

检查 wiki/ 文章格式并自动修复。包含 frontmatter、链接、MOC、Tasks 语法等检查。

### /write — 写作命令

基于 wiki/ 内容撰写文章，新洞察可回流 wiki。

### /research — 研究命令

AI 主动搜索外部信息补充知识空白。

### /health-check — 健康检查命令

全面体检，发现矛盾、空白和结构问题。

### /base — Dataview 视图命令

创建 Dataview 查询视图（任务追踪、阅读列表、时间线等）。

### /review — 复习命令

生成间隔重复复习卡片，深度内化知识。

### /publish — 发布命令

发布到 content/ 用于构建数字花园。

## 编译任务

当我要求你编译时：

1. 扫描 raw/ 和 research/ 中所有文件，识别新增或未处理的内容
2. 对每份新材料，判断它涉及什么主题、子主题、type
3. **知识提炼**：
   - 提取核心洞察、可复用模型、关键数据点
   - 一篇文章涉及多个概念 → 拆分到多篇原子笔记
   - 多份材料讨论同一话题 → 综合整理
   - 特别留意个人笔记中的想法
4. 如果 wiki 中已有对应文章 → 补充和更新
5. 如果是新概念 → 创建新原子笔记，放入多层级文件夹
6. 建立新旧笔记之间的双向链接
7. 如果笔记包含可执行内容，添加 Tasks 语法的待办项
8. 更新 INDEX.md
9. 输出编译报告

**注意**：/compile 不创建 MOC 和主题总结。当原子笔记积累到一定数量（建议 10+ 篇），提示用户执行 /summarize 整理结构。

## 健康检查任务

扫描整个知识库并报告：

- 互相矛盾的内容
- 缺少来源的重要论断
- 过期文章
- 长期草稿
- 孤立文章
- 断链
- 单向链接
- MOC 覆盖率和完整性
- 多层级文件夹结构合理性
- 任务健康度
- 知识分布图
- 值得深挖的线索

## 查询任务

当我提问时：

1. 先查阅 INDEX.md 或 _MOC/知识总览.md 确定相关文章

3. 基于 wiki 内容给出回答
4. 如果信息不足，明确告诉我缺少什么，建议补充什么材料
5. 如果要求保存输出，写入 outputs/
