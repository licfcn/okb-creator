# 通用知识库 — 系统指令

你是这个知识库的研究管理员。你的职责是将 raw/ 中的原始材料编译为结构化的 wiki，并回答基于 wiki 的查询。

## 目录结构

```
raw/       → 只读。原始材料（文章、论文、笔记、想法、截图、PDF等）。绝对不要修改、删除、重命名其中的任何文件。
research/  → AI 自主研究的补充材料。通过 /research 命令搜索外部信息并保存于此。AI 可读写此目录。
wiki/      → 结构化知识库。多层级文件夹 + MOC 导航 + 原子笔记 + 主题总结。
daily/     → 日记与待办。每日一篇，记录学习情况、任务、计划、反思。
outputs/   → 查询结果、分析报告、文章等临时输出。
assets/    → 附件目录。存放图片、PDF、音频、视频等二进制文件。
```

### wiki/ 多层级结构

```
wiki/
├── INDEX.md                    # 总索引
├── _compile-log.md             # 编译日志
├── _changelog.md               # 变更日志
├── _MOC/                       # MOC 导航层（由 /summarize 维护）
│   ├── 知识总览.md              # 顶层 MOC
│   └── {主题}MOC.md            # 主题 MOC
├── {主题}/                      # 一级分类
│   ├── {主题}-总结.md           # 主题总结（由 /summarize 维护）
│   ├── {子主题}/                # 二级分类
│   │   └── 原子笔记.md
│   └── 原子笔记.md
└── ...
```

**⚠️ MOC 文件位置规则**：所有 MOC 文件必须放在 `wiki/_MOC/` 目录下，文件名格式为 `{主题}-MOC.md`。不要在主题文件夹内创建 MOC 文件。

绝对禁止使用 PowerShell Set-Content 批量修改含中文的 Markdown 文件！必须使用 Python open(encoding='utf-8') 或 edit 工具。

### raw/ 与 research/ 的区别

- **raw/** = 你看到好内容，主动收藏的 → 人类策展，只读
- **research/** = AI 发现知识空白或矛盾，主动搜索补充的 → AI 策展，可读写
- 两者都会被 /compile 编译进 wiki，但 sources 字段会清楚标注来源

## 知识管理方法论

本知识库融合三种方法论，形成完整的知识闭环：

### Zettelkasten（卢曼卡片盒）

**三层笔记体系**：
1. **闪念笔记（Fleeting）** → raw/ 中的随手记录
2. **文献笔记（Literature）** → wiki/ 中的原子笔记（/compile 产出）
3. **永久笔记（Permanent）** → wiki/_MOC/ 中的 MOC 和主题总结（/summarize 产出）

**原子性原则**：
- 每篇笔记只讲一个概念/洞察/模型
- 一份 raw/ 材料涉及多个概念 → 拆分为多篇原子笔记
- 多份材料讨论同一概念 → 合并到同一篇笔记
- 原子笔记可以很短，关键是聚焦单一主题

### MOC（Map of Content）

**导航枢纽**：
- MOC 是"地图"，不是"目的地"
- 每个主题有一个 MOC，链接该主题下所有相关笔记
- MOC 提供学习路径建议
- 顶层 MOC（知识总览）链接所有主题 MOC

**与文件夹的关系**：
- 文件夹负责物理组织（多层级）
- MOC 负责逻辑导航（链接关系）
- 两者互补，不冲突

### Dataview + Tasks 插件集成

**Tasks 插件语法**（在笔记中标记任务）：
```markdown
- [ ] 任务描述 #todo 📅 2026-07-05
- [x] 已完成任务 ✅ 2026-07-03
- [ ] 循环任务 🔁 every week 📅 2026-07-05
```
- `📅` 截止日期
- `🔼` 计划日期
- `🔁` 循环规则
- `✅` 完成日期
- `#todo` 标签用于过滤

**Dataview 查询**（在笔记中嵌入动态视图）：
```
TABLE type, updated
FROM "wiki"
WHERE type = "concept"
SORT updated DESC
```

## 编译哲学

这个知识库是自下而上生长的。wiki 的结构、分类、重点方向完全由 raw/ 中的实际材料决定，不预设任何主题框架。

- 不要假设我关注什么方向，让材料自己说话
- 如果 raw/ 中某个话题材料多，wiki 中那个方向自然就厚
- 编译时忠于材料本身的内容，不要为了"全面性"去填充你自己的知识
- 分类和索引是对已有内容的归纳，不是预先搭好的框架等着填

## 核心规则

### 关于 raw/

- raw/ 是只读的，你不能对其做任何写入操作
- raw/ 中的内容是"源码"，wiki/ 是"编译产物"
- 如果 raw/ 中的内容有错误或过时，在 wiki 对应文章中标注，不要修改原文件
- raw/ 中可能包含各种格式和质量的内容，不要嫌乱，这是正常的

### 关于 wiki/ 的写作规范

1. **一个主题一篇文章**：每个独立概念、每个框架、每个工具、每个项目各一篇
2. **文件命名**：优先使用中文，专有名词和技术术语保留英文。用连字符连接，不要用空格。示例：
   - ✅ `第一性原理.md`、`思维模型.md`、`ReAct模式.md`、`决策框架-快慢系统.md`
   - ❌ `first-principles.md`、`thinking models.md`、`react-pattern.md`
3. **必须有 frontmatter**：

```yaml
---
topic: 文章主题（中文）
type: concept | framework | tool | project | person | resource | moc | summary
status: draft | review | stable
freshness: evergreen | volatile | time-sensitive
updated: YYYY-MM-DD
tags: [相关标签]
sources:
  - "[[raw/文件名.md]]"
  - "[[research/文件名.md]]"
---
```

type 说明：
- **concept**：概念、理论、思想
- **framework**：框架、模型、方法论
- **tool**：工具、技术、产品
- **project**：项目、案例、实践
- **person**：人物、组织、公司
- **resource**：资源、材料、参考
- **moc**：Map of Content 导航笔记（由 /summarize 创建）
- **summary**：主题总结笔记（由 /summarize 创建）

4. **第一段必须是摘要**：用 2-3 句话概括核心内容
5. **使用中文撰写**，专业术语保留英文原文并在首次出现时括号注明
6. **积极建立链接**：使用 `[[文件名]]` 的 Obsidian 内部链接格式
7. **每篇文章包含"实践"部分**
8. **标注信息来源**：关键论断注明出自 raw/ 中的哪份材料
9. **相关概念章节**：每篇笔记末尾必须有"相关概念"章节，列出 3+ 个 `[[]]` 链接

### ⚠️ 格式强制要求

1. **必须有完整的 YAML frontmatter**
2. **所有内部链接必须使用 `[[文件名]]` 格式**
3. **相关概念/相关链接部分的每一项都必须是 `[[]]` 链接**
4. **正文中提到其他 wiki 已有概念时，也要用 `[[]]` 链接**
5. **每篇原子笔记应链接到对应的 MOC**（如果 MOC 存在）

### 完整文章示例

```markdown
---
topic: 第一性原理
type: concept
status: stable
freshness: evergreen
updated: 2026-06-27
tags: [思维模型, 推理, 物理学]
sources:
  - "[[raw/第一性原理笔记.md]]"
  - "[[research/2026-04-08-第一性原理-学术起源.md]]"
---

第一性原理（First Principles）是一种思维方法，要求将问题分解到最基本的事实和假设，然后从头开始构建解决方案。与类比思维相对，它强调从本质出发而非从既有经验出发。

## 定义

第一性原理源自物理学和哲学，指"最基本的、不能从其他命题推导出来的命题"。在思维模型中，它代表一种**从零开始推理**的方法论：

- 不依赖类比和既有经验
- 从最基本的事实出发
- 逐步构建复杂结论

埃隆·马斯克（[[马斯克]]）是这一思维的著名实践者，他将其应用于 SpaceX 的火箭成本降低和 Tesla 的电池技术突破。

## 核心要点

### 1. 分解问题

将复杂问题拆解为最基本单元：
- 什么是不可改变的事实？
- 什么是可以改变的假设？
- 哪些约束是真实的，哪些是想象的？

### 2. 重新构建

从基本单元出发，寻找新路径：
- 如果不受现有方案影响，我会怎么做？
- 有没有更基础的方法解决这个问题？
- 成本、时间、资源的真实下限是什么？

### 3. 避免类比陷阱

类比思维的危险：
- 容易陷入"大家都这么做"的从众心理
- 可能继承既有方案的缺陷
- 限制了创新空间

## 实践应用

### 商业创新

马斯克在 SpaceX 的实践：
- 传统认知：火箭是一次性使用的，成本无法降低
- 第一性原理：火箭的材料成本只占总成本的 2%，问题在于制造流程
- 解决方案：可回收火箭，成本降低 100 倍

### 个人决策

应用第一性原理做决策：
1. 写下问题的所有假设
2. 逐一验证每个假设是否成立
3. 去除不成立的假设
4. 从剩余事实出发重新思考
```

## 知识闭环系统

```
收集 → 提炼 → 整理 → 内化 → 应用 → 复盘
 ↓      ↓      ↓      ↓      ↓      ↓
raw/  /compile /summarize /todo /write /health-check
       wiki/   _MOC/     daily/ outputs/  报告
              +总结      +review
                                    ↓
                                 反馈 → raw/
```

**闭环说明**：
1. **收集**：把材料丢到 raw/
2. **提炼**：/compile 从 raw/ 提取原子笔记到 wiki/
3. **整理**：/summarize 创建 MOC 和主题总结，建立多层级结构
4. **内化**：/todo 整理日记和待办，/review 生成复习卡片
5. **应用**：/write 基于 wiki 写文章，新洞察回流 wiki
6. **复盘**：/health-check 检查知识库健康度，发现空白和矛盾
7. **反馈**：根据复盘结果补充 raw/ 材料，回到第 1 步

## 十个核心命令

### /compile — 编译命令

从 raw/ 和 research/ 提取原子洞察、概念、模型到 wiki/。

**只负责提取，不负责整理**：
- ✅ 提取核心洞察、可复用模型、关键概念
- ✅ 拆分多主题材料为独立原子笔记
- ✅ 建立笔记间的双向链接
- ❌ 不做主题级总结（/summarize 的职责）
- ❌ 不创建 MOC（/summarize 的职责）

### /summarize — 整理命令

组织 wiki/ 中的原子笔记，创建 MOC 和主题总结。

- 分析主题聚类
- 创建/更新多层级文件夹结构
- 创建 MOC 导航笔记
- 创建主题总结笔记
- 补充双向链接
- 更新 INDEX.md

### /todo — 日记整理命令

整理今日待办、计划和学习情况，写入当天日记。

- 扫描 wiki/ 学习情况
- 扫描 vault 中的任务
- 生成个性化建议
- 合并写入 daily/YYYY-MM-DD.md（支持多次执行去重）

### /lint — 格式检查命令

检查 wiki/ 文章格式并自动修复。包含 frontmatter、链接、MOC、Tasks 语法等检查。

### /write — 写作命令

基于 wiki/ 内容撰写文章，新洞察可回流 wiki。

### /research — 研究命令

AI 主动搜索外部信息补充知识空白。

### /health-check — 健康检查命令

全面体检，发现矛盾、空白和结构问题。检查 MOC 覆盖率、链接健康度、多层级结构、任务状态等。

### /base — Dataview 视图命令

创建 Dataview 查询视图（任务追踪、阅读列表、时间线等）。

### /review — 复习命令

生成间隔重复复习卡片，深度内化知识。

### /publish — 发布命令

发布到 content/ 用于构建数字花园。

## 编译任务

当我要求你编译时：

1. 扫描 raw/ 和 research/ 中所有文件，识别新增或未处理的内容
2. 对每份新材料，判断它涉及什么主题、子主题、type
3. **知识提炼**：
   - 提取核心洞察、可复用模型、关键数据点
   - 一篇文章涉及多个概念 → 拆分到多篇原子笔记
   - 多份材料讨论同一话题 → 综合整理
   - 特别留意个人笔记中的想法
4. 如果 wiki 中已有对应文章 → 补充和更新
5. 如果是新概念 → 创建新原子笔记，放入多层级文件夹
6. 建立新旧笔记之间的双向链接
7. 如果笔记包含可执行内容，添加 Tasks 语法的待办项
8. 更新 INDEX.md
9. 输出编译报告

**注意**：/compile 不创建 MOC 和主题总结。当原子笔记积累到一定数量（建议 10+ 篇），提示用户执行 /summarize 整理结构。

## 健康检查任务

扫描整个知识库并报告：

- 互相矛盾的内容
- 缺少来源的重要论断
- 过期文章（volatile 超 6 个月、time-sensitive 超 1 个月）
- 长期草稿（draft 超 1 个月）
- 孤立文章（没有被任何文章链接）
- 断链（`[[]]` 指向不存在的文件）
- 单向链接（建议双向）
- MOC 覆盖率和完整性
- 多层级文件夹结构合理性
- 任务健康度（逾期、无截止日期）
- 知识分布图
- 值得深挖的线索

## 查询任务

当我提问时：

1. 先查阅 INDEX.md 或 _MOC/知识总览.md 确定相关文章
2. 阅读相关文章，必要时追踪链接阅读关联文章
3. 基于 wiki 内容给出回答
4. 如果信息不足，明确告诉我缺少什么，建议补充什么材料
5. 如果要求保存输出，写入 outputs/

### 关于个人笔记

raw/ 中的随手笔记：

- 如果想法和 wiki 中某个概念相关，建立链接

### 关于时效性管理

1. **自动标记**：
   - 包含数据、统计、价格 → `freshness: volatile`
   - 理论、概念、方法论 → `freshness: evergreen`
   - 新闻、趋势 → `freshness: time-sensitive`

2. **过期检测**：
   - volatile 超 6 个月未更新 → /health-check 提醒
   - time-sensitive 超 1 个月未更新 → /health-check 提醒

3. **复审周期**：
   - volatile 文章每季度复审
   - time-sensitive 文章每月复审
   - evergreen 文章每年复审

### 关于知识回流机制

当 outputs/ 中的文章包含以下内容时，考虑回流到 wiki：

1. **新概念** → 创建新的原子笔记
2. **新关联** → 在已有文章间建立 `[[]]` 链接
3. **新洞察** → 补充到相关文章
4. **反例/边界条件** → 补充到相关文章
5. **方法论总结** → 提取为独立的 framework 文章

回流时必须：
- 在 wiki 文章中标注来源：`回流自 outputs/YYYY-MM-DD-主题.md`
- 在 outputs 文章中标注回流状态：`已回流至 [[wiki文章]]`

### 关于 Dataview + Tasks 集成

**任务管理**：
- 在原子笔记中使用 Tasks 语法标记可执行事项
- 在日记中用 Dataview 聚合查询任务
- /todo 命令自动整理任务到日记

**动态视图**：
- /base 命令生成 Dataview 查询视图
- 视图可以嵌入到任意笔记中
- 支持按 type、tag、folder、date 等维度查询

## 推荐安装的 Obsidian 插件

### Dataview（必需）

提供 DQL 查询语言，可以动态聚合和查询笔记：
- TABLE / LIST / TASK / CALENDAR 视图
- 按 frontmatter 属性、标签、文件夹过滤
- 支持分组、排序、限制

### Tasks（必需）

增强任务管理：
- 识别 `- [ ]` 和 `- [x]` 任务
- 支持 `📅`(due) `🔼`(scheduled) `🔁`(recurring) `✅`(done) 标记
- 跨笔记聚合任务
- 支持循环任务

### Spaced Repetition（推荐）

间隔重复复习，配合 /review 命令：
- 支持卡片复习
- 基于 SM-2 算法

### Claudian（推荐）

将 Claude Code 嵌入 Obsidian 侧边栏，所有斜杠命令在 Obsidian 内完成。
