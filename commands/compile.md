执行一次编译任务，从 raw/ 和 research/ 中提取原子洞察、概念、模型，生成 wiki/ 中的原子笔记。

## 核心定位

`/compile` 只负责**提取**，不负责**整理**：
- ✅ 提取核心洞察、可复用模型、关键概念、重要数据点
- ✅ 拆分多主题材料为独立原子笔记
- ✅ 建立笔记间的双向链接
- ❌ 不做主题级总结（那是 /summarize 的职责）
- ❌ 不创建 MOC 导航（那是 /summarize 的职责）

**Zettelkasten 分工**：
- raw/ = 闪念笔记 + 源材料
- /compile → wiki/ 原子笔记（文献笔记层）
- /summarize → wiki/_MOC/ 导航 + 主题总结（永久笔记层）

## wiki/ 多层级文件夹结构

wiki/ 支持最多 3 层文件夹深度：

```
wiki/
├── INDEX.md                    # 总索引
├── _compile-log.md             # 编译日志
├── _changelog.md               # 变更日志
├── _MOC/                       # MOC 导航层（由 /summarize 维护）
│   ├── 知识总览.md
│   └── {主题}MOC.md
├── {主题}/                     # 一级分类
│   ├── {主题}-总结.md          # 主题总结（由 /summarize 维护）
│   ├── {子主题}/               # 二级分类
│   │   ├── 原子笔记1.md
│   │   └── 原子笔记2.md
│   └── 原子笔记3.md
└── ...
```

**多层级规则**：
1. 编译时根据材料内容判断所属主题和子主题
2. 如果该主题文件夹不存在则创建
3. 子主题文件夹在父主题文件夹下创建
4. 文件夹名称用中文，专有名词保留英文
5. 一篇笔记只放在一个最具体的文件夹中
6. `[[]]` 链接中使用文件名即可，Obsidian 会自动解析路径

## 编译来源

编译时扫描两个目录：

1. **raw/**：用户手动收集的原始材料（只读，不可修改）
2. **research/**：AI 通过 /research 命令搜索整理的补充材料（可读写）

wiki 文章的 sources 字段必须使用双链 `[[]]` 格式标注来源，这样 Obsidian 能建立反向链接，让原始材料也能看到被哪些笔记引用：

```yaml
sources:
  - "[[raw/马斯克工作法则.md]]"
  - "[[research/2026-04-08-第一性原理-学术起源.md]]"
```

注意：使用 YAML 块列表格式（`- ` 开头），每项用双引号包裹，保留 `.md` 扩展名。

对于 research/ 中的文件，编译完成后将其 frontmatter 中的 `status` 改为 `compiled`。

## 编译日志机制

wiki/ 目录下维护 `_compile-log.md`，追踪每个文件的编译状态。

### 编译前：扫描并对比

1. 读取 `wiki/_compile-log.md`（不存在则创建）
2. 列出 raw/ 和 research/ 中所有文件（排除 .gitkeep）
3. 对比日志，分为三类：
   - **🆕 新文件**：日志中不存在 → 需要编译
   - **📝 已修改**：内容或大小变化 → 需要重新编译
   - **✅ 无变化**：未变化 → 跳过
4. 报告：raw/ 中 X 新、Y 改、Z 不变；research/ 中同理
5. 仅对新文件和已修改文件执行编译

### 增量更新

新文件编译时检查与已有 wiki 文章的关联：

- 主题重叠 → **合并更新**已有文章，不创建重复
- 合并时保留原结构，补充新信息，更新 `updated` 日期，追加 sources
- 新概念关联 → 补充 `[[]]` 链接

## 原子笔记写作规范

### 原子性原则

每篇 wiki 文章应该是一个**原子笔记**：
- 一个概念/洞察/模型 = 一篇文章
- 如果一份 raw/ 材料涉及多个概念 → 拆分为多篇原子笔记
- 如果多份材料讨论同一概念 → 合并到同一篇原子笔记
- 原子笔记可以很短（哪怕只有几段），关键是聚焦单一主题

### frontmatter 规范

```yaml
---
topic: 文章主题（中文）
type: concept | framework | tool | project | person | resource
status: draft | review | stable
freshness: evergreen | volatile | time-sensitive
updated: YYYY-MM-DD
tags: [相关标签]
sources:
  - "[[raw/文件名.md]]"
  - "[[research/文件名.md]]"
---
```

新增 type 说明：
- **concept**：概念、理论、思想
- **framework**：框架、模型、方法论
- **tool**：工具、技术、产品
- **project**：项目、案例、实践
- **person**：人物、组织、公司
- **resource**：资源、材料、参考

注：`moc` 和 `summary` 类型由 /summarize 命令使用，/compile 不创建这两种类型。

### 双向链接规范

原子笔记必须积极建立 `[[]]` 链接：

1. **正文链接**：提到其他 wiki 已有概念时，用 `[[文件名]]`
2. **相关概念章节**：每篇笔记末尾必须有"相关概念"章节，列出 3+ 个 `[[]]` 链接
3. **来源链接**：sources 字段使用 `[[]]` 双链标注来源文件，Obsidian 会自动建立反向链接
4. **跨主题链接**：如果概念跨主题，主动链接到另一主题的笔记

### 任务标记（Tasks 插件集成）

如果原子笔记中包含可执行的待办事项，使用 Tasks 插件语法：

```markdown
## 实践任务

- [ ] 实践第一性原理分析一个实际问题 #todo 📅 2026-07-05
- [ ] 记录三个可以应用此概念的决策场景 #todo
```

语法说明：
- `- [ ]` 未完成，`- [x]` 已完成
- `#todo` 标签用于 Dataview/Tasks 查询过滤
- `📅 YYYY-MM-DD` 表示截止日期
- `🔼 YYYY-MM-DD` 表示计划日期
- `🔁 every week` 表示循环任务

### Dataview 查询嵌入

原子笔记可以在末尾嵌入 Dataview 查询，展示相关笔记：

```markdown
## 相关笔记

```dataview
TABLE type, updated
FROM "wiki"
WHERE contains(file.tags, "思维模型") AND file.name != this.file.name
SORT updated DESC
```
```

## 编译步骤

1. 执行扫描对比流程（同时扫描 raw/ 和 research/）
2. 对需要编译的新文件：
   a. 判断内容涉及哪些主题、子主题、type
   b. **拆分多主题材料**：一份 raw/ 涉及多个概念 → 拆分多篇原子笔记
   c. 检查是否和已有 wiki 文章主题重叠 → 增量更新或新建
   d. 新建时放入对应的多层级文件夹（不存在则创建）
3. 每篇原子笔记必须包含完整 frontmatter 和 `[[]]` 链接
4. 建立新旧笔记之间的双向链接
5. 如果笔记包含可执行内容，添加 Tasks 语法的待办项
6. 更新 wiki/INDEX.md（反映多层级结构）
7. 更新 wiki/_compile-log.md
8. 将 research/ 中已编译文件的 status 改为 `compiled`
9. 在 wiki/_changelog.md 顶部追加编译记录
10. 输出编译报告

## 注意

- 不要修改 raw/ 中的任何文件
- research/ 中的文件可以修改 status 字段
- 不要跳过 frontmatter
- 所有相关概念引用必须使用 `[[文件名]]` 格式
- 文件命名中文优先，专有名词保留英文
- 不要创建 MOC 或总结笔记（那是 /summarize 的职责）
- 一份材料涉及多个主题时，必须拆分为多篇原子笔记
